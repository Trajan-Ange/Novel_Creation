# 小说创作管理系统 — 更新需求文档

> 基于 v0.0.1 全项目代码审查，整理缺陷、优化点和功能缺口。

---

## 一、安全缺陷（需优先修复）

### 1.1 明文 API Key 存储 [CRITICAL]

- **现状：** `config.json` 包含明文 API Key，已通过 `.gitignore` 排除出版本库，但文件中仍以明文持久化
- **影响：** 若文件被误分享或备份泄露，API Key 直接暴露
- **建议方案：**
  - 支持从环境变量 `NOVEL_LLM_API_KEY` 读取
  - 优先级：环境变量 > config.json
  - 在 `config.example.json` 中注明环境变量用法
  - 长期可考虑使用系统密钥链（keyring）

### 1.2 路径遍历漏洞 [HIGH]

- **位置：** `app/storage/file_manager.py` — `delete_project()` 方法
- **问题：** `delete_project` 未调用 `_validate_name()` 进行名称校验，攻击者可传入 `../` 类路径删除 projects 目录外的文件
- **修复：** 在 `delete_project` 中先调用 `_validate_name(name)`，然后使用 `os.path.realpath` 校验最终路径仍在 projects 目录内

### 1.3 HTTP 状态码不规范 [HIGH]

- **现状：** 所有 API 端点出错时返回 HTTP 200 + `{"success": false, "error": "..."}`
- **影响：** 前端无法根据状态码区分错误类型，调试困难
- **建议方案：**
  - 资源不存在 → 404
  - 参数校验失败 → 422
  - 资源冲突（重名）→ 409
  - 服务器内部错误 → 500
  - 保持 `{"success": false, "error": "..."}` 响应体格式不变

### 1.4 配置文件非原子写入 [MEDIUM]

- **位置：** `app/services/llm.py:save_config()`、`file_manager.py` 中多处写操作
- **问题：** 直接覆盖写入，写入过程中若崩溃则文件损坏
- **修复：** 采用"写临时文件 + 原子重命名"模式

## 二、功能缺陷

### 2.1 伏笔 ID 生成碰撞风险 [HIGH]

- **位置：** `app/skills/knowledge_sync.py` 第 221 行
- **问题：** ID 格式 `FB-{chapter}-{len(new_fbs)}` 不唯一，不同章节可能生成相同 ID
- **修复：** 加入时间戳或 UUID 短码，如 `FB-{chapter}-{timestamp}-{seq}`

### 2.2 删除项目缺少名称校验 [HIGH]

- 同 1.2，见安全缺陷

### 2.3 LLM 调用无重试机制 [HIGH]

- **位置：** `app/services/llm.py`
- **问题：** 网络抖动、API 限流等瞬时故障直接导致整个工作流失败
- **建议方案：**
  - 添加指数退避重试（最多 3 次）
  - 仅重试可恢复错误（429、5xx、网络超时）
  - 重试前记录日志

### 2.4 SSE 流式传输不检测客户端断开 [HIGH]

- **位置：** `app/api/chapters.py`、`app/api/settings.py`
- **问题：** 用户关闭页面后，后端继续生成内容，浪费 API Token
- **修复：** 在 SSE 生成循环中检查 `await request.is_disconnected()`

### 2.5 大纲树加载存在 N+1 查询 [MEDIUM]

- **位置：** `app/static/js/components/outline-tree.js` — `loadOutlineTree()`
- **问题：** 加载卷列表后，在循环内逐个请求每个卷的章节列表（串行），N 个卷需要 N+1 次请求
- **修复：** 使用 `Promise.all` 并行请求所有卷的章节列表

### 2.6 人物名称提取逻辑脆弱 [MEDIUM]

- **位置：** `app/skills/character_design.py`
- **问题：** 从 `instruction.split("，")[0]` 提取角色名，用户输入"创建角色：张三，他是剑客"时提取到的是"创建角色：张三"
- **修复：** 使用正则匹配"创建|新增|添加"后的名称部分

### 2.7 `_split_characters` 仅识别 h1 标题 [MEDIUM]

- **位置：** `app/api/settings.py`
- **问题：** 按 `^# (.+)$` 正则切分角色，若 LLM 输出 h2 标题则全部写入同一文件
- **修复：** 同时支持 h1/h2 切分，或使用更宽松的匹配

### 2.8 知识同步版本号粒度过粗 [MEDIUM]

- **位置：** `app/skills/knowledge_sync.py`
- **问题：** 每次同步递增所有版本号，即使只改动了角色设定，世界设定版本也会 +1
- **修复：** 仅递增实际发生变更的模块版本号

### 2.9 上下文硬截断可能切断关键信息 [MEDIUM]

- **位置：** `app/api/settings.py` — `_get_relevant_context()`
- **问题：** 硬编码截断长度（2000、800、1500 字符），可能在句子中间截断
- **建议方案：**
  - 按段落边界截断
  - 添加截断标记 `[...已截断]`
  - 长期引入 token 计数精确控制上下文长度

### 2.10 连接测试 temperature=0 兼容性问题 [LOW]

- **位置：** `app/api/config.py`
- **问题：** 部分模型拒绝 `temperature=0`
- **修复：** 改为 `temperature=0.1`

### 2.11 时间线拆分依赖精确标题文本 [LOW]

- **位置：** `app/skills/timeline.py`
- **问题：** 按 `# 故事时间线` 精确字符串拆分，LLM 输出格式稍有变化即拆分失败
- **修复：** 使用更宽松的正则匹配，或要求 LLM 输出结构化 JSON

## 三、代码质量问题

### 3.1 死代码清理

| 文件/函数 | 说明 |
|-----------|------|
| `app/api/settings_chat.py` | 完整模块，未被 `main.py` 注册路由，功能已被 `settings.py` 中的 `chat-generate` 端点替代 |
| `app/core/orchestrator.py:create_project_workflow()` | 未被任何 API 端点调用，前端使用自己的向导流程 |
| `app/core/orchestrator.py:apply_extracted_lore()` | 未被任何 API 端点调用 |
| `outline-tree.js:generateOutlineLegacy()` | 标记为"向后兼容"但无引用 |

### 3.2 重复代码

| 重复内容 | 位置 1 | 位置 2 |
|----------|--------|--------|
| 访谈提示词 `INTERVIEW_BASE` / `INTERVIEW_GUIDES` | `app/api/settings.py` | `app/api/settings_chat.py` |
| `escapeHtml()` 函数 | `settings-editor.js` | `settings-chat.js` |
| SSE `send()` 闭包模式 | `chapters.py` | `settings.py`（多处） |

- **建议：** 将公共工具提取到 `app/api/utils.py` 和 `app/static/js/utils/` 下

### 3.3 未使用的依赖

- `aiofiles>=23.2.0` — 在 `requirements.txt` 中声明但全项目无任何 import
- **修复：** 移除或实际使用异步文件 I/O

### 3.4 错误信息泄露

- **问题：** 大量 `except Exception as e: return {"success": False, "error": str(e)}` 将 Python 内部错误信息暴露给前端
- **建议：** 生产环境应返回通用错误消息，详细信息写入日志

## 四、架构与性能优化

### 4.1 版本快照机制缺失 [MEDIUM]

- **现状：** `项目状态.json` 记录了版本号，`版本历史/` 目录已创建但从未写入内容
- **需求：** 每次知识同步后，将被修改的文件快照保存到 `版本历史/`，支持回退

### 4.2 缺少 Token 计数与上下文窗口管理 [MEDIUM]

- **问题：** 所有 LLM 调用的上下文组装使用硬编码截断，不感知实际 token 数
- **风险：** 角色/章节增多后可能超出模型上下文窗口，导致静默截断
- **建议方案：**
  - 引入 `tiktoken` 库进行精确 token 计数
  - 在 `chat_with_context()` 中添加上下文窗口预算管理
  - 按优先级分配 token：当前章节大纲 > 角色设定 > 世界设定 > 历史章节

### 4.3 文件锁与并发安全 [MEDIUM]

- **问题：** 当前无任何文件锁机制，若两个请求同时操作同一项目文件，数据可能损坏
- **建议方案：**
  - 引入项目级 `asyncio.Lock`，按项目 ID 分片
  - 或使用文件系统锁（`portalocker` / `fcntl`）

### 4.4 项目名称校验字符覆盖不足 [LOW]

- **位置：** `app/storage/file_manager.py:_validate_name()`
- **问题：** Unicode 范围 `一-鿿` 仅覆盖基本汉字块，缺少扩展区汉字和常用符号（括号、点号等）
- **修复：** 使用 `\w` 配合 Unicode flag，或扩大字符范围

### 4.5 全文搜索性能 [LOW]

- **位置：** `app/storage/file_manager.py:search()`
- **问题：** 每次搜索遍历所有 .md 文件做内存中关键词匹配，项目文件多时性能差
- **建议：** 长期可引入简单索引（如 whoosh）或使用 `ripgrep` 子进程

## 五、前端优化

### 5.1 SSE 客户端无重连机制 [MEDIUM]

- **位置：** `app/static/js/utils/sse.js`
- **问题：** 连接断开后需手动刷新页面
- **修复：** 添加自动重连（指数退避，最多 3 次），并通过 `onDisconnect` 回调通知用户

### 5.2 CDN 脚本缺少 SRI 完整性校验 [LOW]

- **位置：** `app/static/index.html`
- **问题：** `marked.js` 通过 CDN 加载，无 `integrity` 属性
- **修复：** 添加 SRI hash

### 5.3 仪表盘数据不自动刷新 [LOW]

- **位置：** `app/static/js/components/foreshadowing.js:renderDashboard()`
- **问题：** 撰写章节后返回仪表盘，显示的是旧数据
- **修复：** 在导航回仪表盘时重新拉取数据

### 5.4 文本编辑器体验差 [LOW]

- **位置：** `app/static/js/components/settings-editor.js`
- **问题：** 手动编辑时替换为不可调整大小的 textarea，长文档编辑体验差
- **建议：** 使用 Monaco Editor 或 CodeMirror 轻量版，或至少让 textarea 可拖拽调整大小

## 六、工程质量

### 6.1 零测试覆盖 [HIGH]

- **现状：** 项目无任何单元测试、集成测试或端到端测试
- **建议方案：**
  - 后端：`pytest` + `httpx`（FastAPI TestClient）覆盖 API 端点
  - 核心逻辑：FileManager、LLMService 的单元测试
  - 前端：至少添加关键用户流程的手动测试清单

### 6.2 缺少结构化日志 [MEDIUM]

- **现状：** 使用 `print` 或无语录
- **建议：** 引入 `logging` 模块，按级别输出，支持文件持久化

### 6.3 `test_config.json` 定位不清 [LOW]

- **问题：** 文件存在但用途不明，值为占位符，且已被 `.gitignore` 排除
- **建议：** 删除或明确为 CI 测试配置模板
