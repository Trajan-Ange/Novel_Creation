## 一、项目概述

### 1.1 项目背景

长篇小说创作是一个复杂的系统工程，涉及世界观构建、人物塑造、剧情设计、细节一致性维护等多维度工作。传统创作方式存在以下痛点：

- **设定散乱**：世界观、人物、时间线等设定分散，难以统一管理
- **一致性难维护**：随着章节增加，容易出现前后矛盾
- **信息检索困难**：写作时难以快速查找相关设定和历史剧情
- **伏笔易遗漏**：埋下的伏笔容易忘记回收
- **二创重复劳动**：基于已有世界观创作时，需重复整理基础设定

### 1.2 项目目标

构建一个智能化的小说创作与管理系统，实现：

1. **多层次创作支持**：从全书大纲到章节正文的完整创作链
2. **结构化知识管理**：世界观、人物、时间线等创作依据的系统化管理
3. **自动化知识同步**：章节生成后自动更新所有创作依据
4. **智能辅助能力**：记忆检索、风格管理、伏笔追踪
5. **二创友好**：支持采集已有世界观，减少重复劳动

### 1.3 核心创新点

- **自我更新机制**：创作依据随章节内容自动演进
- **伏笔全流程管理**：埋下→追踪→提醒→回收的完整闭环
- **世界观采集与蒸馏**：联网搜索已有作品设定，结构化输出
- **交互式创作**：用户深度参与每章创作，也可选择自动生成

## 二、系统架构

### 2.1 架构模式

采用 **智能体 + 多技能** 的混合架构：

```
┌─────────────────────────────────────────────────────────────┐
│                    主控智能体：小说创作助手                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  • 工作流调度与状态管理                                │   │
│  │  • 创作依据的协调调用                                  │   │
│  │  • 用户交互与意图识别                                  │   │
│  │  • 自动触发自我更新流程                                │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                         技能体系                             │
│                                                             │
│  第一层：创作依据管理                                        │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌────────┐ │
│  │世界设定管理  │ │人物设定管理  │ │时间线管理   │ │关系管理│ │
│  └─────────────┘ └─────────────┘ └─────────────┘ └────────┘ │
│                                                             │
│  第二层：内容生成与管理                                      │
│  ┌─────────────┐ ┌─────────────┐                            │
│  │大纲生成管理  │ │章节正文生成  │                            │
│  └─────────────┘ └─────────────┘                            │
│                                                             │
│  第三层：核心引擎                                            │
│  ┌─────────────┐ ┌─────────────┐                            │
│  │知识同步引擎  │ │写作智能辅助  │                            │
│  └─────────────┘ └─────────────┘                            │
│                                                             │
│  第四层：辅助工具                                            │
│  ┌─────────────┐                                            │
│  │世界观采集蒸馏│                                            │
│  └─────────────┘                                            │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 架构设计原则

1. **职责分离**：智能体负责调度，技能负责执行
2. **高内聚低耦合**：每个技能专注于单一核心能力
3. **可扩展性**：技能可独立迭代，可灵活组合
4. **可复用性**：技能可被其他创作类智能体调用

## 三、主控智能体设计

### 3.1 核心职责

| 职责 | 描述 |
|------|------|
| 工作流调度 | 根据用户意图选择并执行对应工作流 |
| 状态管理 | 维护项目进度、创作模式、待回收伏笔等状态信息 |
| 技能协调 | 按正确顺序调用技能，传递必要上下文 |
| 用户交互 | 询问、确认、呈现结果、收集反馈 |
| 自动更新触发 | 章节生成后自动调用知识同步引擎 |

### 3.2 维护的状态信息

```yaml
项目状态:
  项目名称: [小说名称]
  创作类型: [原创/二创]
  源作品信息: [若是二创，记录源作品名称]
  当前进度:
    当前卷: [第X卷]
    当前章: [第X章]
  阶段: [初次创建/大纲创作/正文创作/修改中]
  创作模式: [交互式/自动生成]
  待回收伏笔列表: [伏笔ID列表]
  最近更新时间: [时间戳]

创作依据状态:
  世界设定版本: [v1.0]
  人物设定版本: [v1.2]
  时间线版本: [v1.1]
  关系图谱版本: [v1.0]
  风格指南版本: [v1.0]
```

### 3.3 用户意图识别

智能体需识别以下用户意图：

| 意图类型 | 触发示例 | 执行工作流 |
|---------|---------|-----------|
| 初次创建 | "我想写一部修仙小说" | 工作流1 |
| 世界观采集 | "我想写《原神》的同人" | 工作流0 + 工作流1 |
| 继续创作 | "继续写下一章" | 工作流2 |
| 自动生成 | "自动生成剩余章节" | 工作流2（自动模式） |
| 修改设定 | "把主角的性格改一下" | 工作流3 |
| 直接给出章节 | "这是第10章的内容..." | 工作流3 |
| 查询信息 | "主角在第5章是什么状态？" | 调用记忆检索 |
| 伏笔管理 | "有哪些伏笔还没回收？" | 调用伏笔管理 |

#### 4.2.2 novel-chapter-write（章节正文生成）

**核心能力**：
- 基于章节大纲生成正文
- 保持风格一致性
- 融入创作依据
- 处理伏笔埋设

**输入**：
- 章节大纲
- 创作依据（人物设定、世界设定、时间线等）
- 风格指南
- 伏笔提示（待埋/待回收）

**输出**：
- 正文/第X卷/第X章.md

**生成约束**：
1. **优先级原则**：用户直接指令 > 创作依据各文件 > AI自主生成
2. **风格一致**：符合已建立的风格指南
3. **设定遵循**：不违背已有的世界设定和人物设定
4. **伏笔处理**：按大纲要求埋设或回收伏笔

### 4.3 第三层：核心引擎（2个技能）

#### 4.3.1 novel-knowledge-sync（知识同步引擎）⭐ 核心

**核心能力**：
- 章节内容解析与信息提取
- 智能分发更新任务
- 一致性检查与冲突标记
- 更新摘要生成

**触发时机**：
- 每章生成完成后自动触发
- 用户修改设定/大纲/章节后触发
- 用户直接给出章节后触发

**执行流程**：
```
输入：章节正文
↓
1. 内容解析
   - 提取实体：新出现的人物、地点、物品
   - 提取事件：发生了什么，时间节点
   - 提取变化：人物状态变化、能力变化、关系变化
   - 提取伏笔：潜在的伏笔线索
↓
2. 识别更新目标
   - 新人物 → novel-character-design
   - 人物状态变化 → novel-character-design
   - 新区域/势力 → novel-world-design
   - 事件发生 → novel-timeline
   - 关系变化 → novel-relationship
   - 伏笔埋设 → novel-writing-assist
↓
3. 调用对应技能执行更新
↓
4. 一致性检查
   - 检查新内容是否与已有设定冲突
   - 标记需要用户确认的矛盾点
↓
5. 生成更新摘要
   输出：更新摘要报告
```

**更新摘要格式**：
```markdown
# 第X章 更新摘要

## 新增内容
- 新人物：[角色名]
- 新区域：[地名]
- 新物品：[物品名]

## 状态更新
### 角色A
- 能力：新增 [能力C]
- 性格：开朗 → 阴沉

### 角色B
- 状态：受伤（轻度）

## 关系更新
- 角色A - 角色D：陌生 → 仇敌

## 时间线更新
- 第X天：[事件B]

## 伏笔埋设
- [伏笔描述]

## 冲突警告
⚠️ 角色A在第3章已学会剑法，本章又描写为"第一次握剑"，请确认是否需要调整。
```

#### 4.3.2 novel-writing-assist（写作智能辅助）⭐ 核心

**整合四大能力**：

##### 能力1：记忆检索

**查询类型**：
| 查询类型 | 示例 | 数据源 |
|---------|------|--------|
| 人物历史状态 | "角色A在第15章是什么状态？" | 人物设定 |
| 剧情关联 | "角色B之前和谁有过冲突？" | 人物设定 + 故事时间线 |
| 时间查询 | "距离上一次大战过了多久？" | 故事时间线 |
| 设定查询 | "这个世界有什么修炼等级？" | 世界设定 |
| 关系查询 | "角色C和角色D是什么关系？" | 人物关系 |

**实现方式**：
- 基于语义检索（memory_search）
- 支持模糊查询和多条件组合

---

##### 能力2：风格管理

**风格要素**：
```yaml
风格指南:
  叙事视角: [第一人称/第三人称限制/第三人称全知]
  语言风格:
    整体基调: [古风/现代/文艺/口语化]
    词汇选择: [典雅/通俗/混合]
    句式特点: [长句为主/短句为主/长短结合]
  对话特色:
    风格: [简洁/絮叨/文雅/粗犷]
    口头禅/特色表达: [如有]
  描写偏好:
    侧重: [心理描写/场景描写/动作描写]
    比喻风格: [常用比喻类型]
  节奏控制:
    整体节奏: [紧凑/舒缓/张弛有度]
    信息密度: [高/中/低]
```

**风格采集**：
- 从已生成章节中自动提取风格特征
- 生成风格指南初稿
- 用户可修改完善

**风格检测**：
- 检测当前章节风格是否偏离
- 提供修正建议
- 生成风格一致性评分

##### 能力3：伏笔管理

**伏笔全流程**：

```
【埋下阶段】
输入：章节内容/用户指令
↓
1. 识别潜在伏笔
   - 用户明确要求的伏笔
   - AI判断值得埋设的伏笔
↓
2. 记录伏笔信息
   - 伏笔ID：FB-[章节号]-[序号]
   - 埋设位置：第X章
   - 伏笔内容：[描述]
   - 涉及人物/物品：[相关实体]
   - 预期回收方式：[用户确认或AI建议]
   - 状态：待回收
↓
3. 保存到伏笔清单

【追踪阶段】
每次创作时：
1. 检索所有"待回收"状态的伏笔
2. 判断当前章节是否适合回收
   - 时间/剧情是否合适
   - 是否有足够的铺垫
3. 若适合 → 提示用户
4. 若不适合 → 继续追踪

【回收阶段】
输入：用户确认回收方式
↓
1. 在正文中处理伏笔
2. 更新伏笔状态：已回收
3. 记录回收位置：第X章
4. 记录回收方式：[具体描述]
```

**伏笔文档结构**：
```markdown

##### 能力4：风格检测

**检测维度**：
1. **语言一致性**：用词风格、句式结构是否符合既定风格
2. **人物一致性**：人物对话风格、行为模式是否一致

## 七、用户交互设计

### 7.1 交互原则

1. **渐进式引导**：不一次性要求用户提供所有信息，而是在需要时询问
2. **默认交互式**：默认让用户参与每章创作，而非全自动生成
3. **清晰的反馈**：每一步都给用户明确的反馈和选择
4. **容错性**：允许用户随时修改，支持回滚

       - 建议：可在第一卷末或第二卷回收

    当前章节：第6章
    建议在写作时考虑回收伏笔。
```

## 八、技术实现要点

### 8.1 技能通信机制

- 技能之间通过文件系统通信
- 主控智能体通过调用技能（skill_load）触发执行
- 技能执行完成后返回结果文件路径

### 8.2 状态管理

- 项目状态保存在 项目状态.json
- 智能体每次启动时读取状态文件
- 状态变更后立即写入

### 8.3 记忆检索实现

- 利用 memory_search 工具进行语义检索
- 创作依据文件作为检索源
- 支持多条件组合查询

### 8.4 自我更新机制

```
触发条件：章节正文生成完成 / 用户修改 / 用户直接给出章节
↓
调用技能：novel-knowledge-sync
↓
执行步骤：
1. 读取章节内容
2. 使用LLM解析提取信息
3. 根据信息类型调用对应技能
4. 各技能更新对应的设定文件
5. 生成更新摘要
```

### 8.5 一致性检查

```
检查时机：
- 自我更新时
- 用户修改设定时

检查方法：
1. 提取变更内容的关键信息
2. 检索所有已生成章节和相关设定
3. 使用LLM判断是否存在冲突
4. 标记冲突点并提示用户
```

### 8.6 伏笔管理实现

```
伏笔存储：
- 伏笔清单.md：索引文件，记录所有伏笔
- 伏笔详情/：每个伏笔一个详细文件

追踪机制：
- 每次生成章节前，检索"待回收"状态的伏笔
- 根据当前剧情进度判断是否适合回收
- 提供回收建议

回收机制：
- 用户确认回收方式后，在正文生成时融入
- 更新伏笔状态为"已回收"
- 记录回收位置和方式
```
